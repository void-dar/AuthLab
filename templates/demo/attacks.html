💉 SQL Injection Attack

Classic SQLi bypass: ' OR '1'='1 makes the WHERE clause always true, bypassing authentication.

❌ Vulnerable Query

query = f"""
SELECT * FROM users 
WHERE username='{username}' 
AND password='{password}'
"""
# Input: username = ' OR '1'='1'--
# Result: SELECT * FROM users 
#         WHERE username='' OR '1'='1'
# Always returns a row → auth bypass!

✅ Secure (Parameterized)

query = """
SELECT * FROM users 
WHERE username = ? 
AND password_hash = ?
"""
cursor.execute(query, (username, hash))
# Input is data, never executed as SQL
# SQLModel/SQLAlchemy does this for you

🔨 Brute Force Attack

Simulate rapid login attempts to demonstrate rate limiting and account lockout defenses.

Attack: Attacker tries thousands of password combinations automatically.
Defense: Rate limiting (5 req/min) + account lockout after 5 failed attempts + exponential backoff.

🔍 IDOR — Insecure Direct Object Reference

IDOR occurs when a server exposes internal IDs without checking if the requester is authorized to access them.

❌ Vulnerable

GET /api/users/profile/5
# No auth check!
# User 3 can read user 5's data
# Just increment the ID to access anyone

✅ Secure

GET /api/users/profile/5
# Server checks: current_user.id == 5
# OR current_user.role == "admin"
# Otherwise: 403 Forbidden

🪙 JWT Algorithm Confusion Attack

Attacker changes the algorithm in the JWT header to "none" to bypass signature verification — only possible if the server doesn't validate the algorithm.

The "alg: none" attack: If a server accepts any algorithm, an attacker can change "alg": "HS256" to "alg": "none", modify the payload to escalate privileges, and drop the signature. A vulnerable server accepts the unsigned token!
