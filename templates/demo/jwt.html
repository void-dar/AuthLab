What is a JWT?

A JSON Web Token is a compact, URL-safe token with three Base64URL-encoded parts: Header . Payload . Signature

The server signs the token with a secret key. If anyone modifies the payload, the signature check fails — but the payload is NOT encrypted, only signed. Never put secrets in JWT payloads!

Refresh Token Flow

Why two tokens? Access tokens are short-lived (15–30 min) to limit damage if stolen. Refresh tokens are long-lived and used only to issue new access tokens. This way, even if an access token is intercepted, it expires quickly.
1️⃣
Login → Server issues access_token (15min) + refresh_token (7 days)
↓
2️⃣
Use API → Send access_token in cookie. Server validates signature + expiry.
↓
3️⃣
Access token expires → Client sends refresh_token to /api/auth/refresh
↓
4️⃣
New tokens issued → Old refresh token invalidated (token rotation). New pair issued.
↓
🚨
Theft detection: If old refresh token used again → all tokens for that user revoked (reuse detection)

🔐 JWT Security Best Practices

❌ Never Do

  • Store sensitive data in payload
  • Use "alg: none" (algorithm confusion)
  • Store JWT in localStorage (XSS risk)
  • Use very long expiration
  • Skip signature verification

✅ Always Do

  • Use HttpOnly cookies for storage
  • Short expiration (15–30 min)
  • Validate algorithm explicitly (HS256)
  • Implement refresh token rotation
  • Include jti (JWT ID) for revocation
