🏴 Cookie Security Flags

HttpOnly

JavaScript cannot read this cookie. Protects against XSS attacks stealing session tokens.

Set-Cookie: session=abc; HttpOnly

Secure

Cookie only sent over HTTPS. Prevents token interception over HTTP (network sniffing).

Set-Cookie: session=abc; Secure

SameSite=Strict

Cookie NOT sent on cross-site requests. Defeats CSRF attacks — attacker's page can't trigger requests with your cookies.

Set-Cookie: session=abc; SameSite=Strict

SameSite=Lax

Sent on top-level navigations (clicking links) but NOT on subresource requests. Balance between security and usability.

Set-Cookie: session=abc; SameSite=Lax

No Flags ❌

Cookie readable by JS (XSS), sent over HTTP (sniffable), sent cross-site (CSRF). Never do this for auth cookies.

Set-Cookie: session=abc ← vulnerable!

✅ AuthLab Cookie

This app uses all recommended flags.

Set-Cookie: access_token=...; HttpOnly; Secure; SameSite=lax; Path=/

🔄 Session Lifecycle

1️⃣
Login — User submits credentials. Server verifies password hash.
↓
2️⃣
Session Created — Server generates random session ID, stores {session_id → user_data} in DB/Redis.
↓
3️⃣
Cookie Set — Session ID sent to browser as HttpOnly cookie.
↓
4️⃣
Requests — Browser sends cookie automatically. Server looks up session ID, retrieves user.
↓
5️⃣
Logout / Expiry — Server deletes session record. Old session ID is now invalid immediately.
Key difference from JWT: In session-based auth, the server controls session validity. Logout is instant. With JWT, you must wait for the token to expire — unless you implement a blocklist (which adds statefulness back).

⚠️ Attack: Session Fixation

An attacker sets a known session ID in the victim's browser before they log in, then uses that same ID to hijack the session after the victim authenticates.

❌ Vulnerable

# Server reuses pre-login session ID
session_id = request.cookies.get("session") 
if session_id is None:
    session_id = generate_id()
# Attacker sets session_id=known_value
# After victim logs in: attacker has access

✅ Secure (AuthLab)

# Always generate NEW session ID on login
# Old session IDs are invalidated
session_id = secrets.token_urlsafe(32)
db.create_session(user_id, session_id)
# Attacker's known ID no longer valid
